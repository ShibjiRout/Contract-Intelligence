CODEBASE ARCHITECTURE REPORT
Project: Contract Intelligence Platform
Scope date: 2026-05-12
Scan scope: source code only. Markdown files and Claude/agent scaffolding were excluded from the review.

1. Executive Summary

This repository is a full-stack contract review platform with:
- A FastAPI backend in `contracts_platform`
- A React + Vite frontend in `frontend`
- Celery workers for asynchronous document processing
- MongoDB for operational contract/clause state
- PostgreSQL for playbook rules and rule metadata
- Qdrant for clause and playbook semantic search
- Neo4j for contract/party/clause relationship history
- Azure Blob Storage for uploaded files
- Azure File Share for OCR text persistence
- Redis for rate limiting, progress events, OCR caching, and task transport/backend
- OpenAI and Azure Document Intelligence for extraction and recommendation generation

The product goal is to take uploaded contracts, extract clauses, compare them against internal standards, and present lawyers with AI-assisted review recommendations and approval workflows.

3. Runtime Architecture

3.1 Synchronous request path

The FastAPI app starts in `contracts_platform/api/main.py`.
Startup behavior:
- Configures structlog logging
- Configures optional Azure OpenTelemetry tracing
- Registers CORS, request logging middleware, and Redis-backed rate limiting middleware
- Registers router groups for admin, auth, contracts, clauses, users, and webhooks
- Exposes `/health`
- Exposes `/ws/contracts/{contract_id}` for live progress updates

3.2 Asynchronous processing path

The main business flow is not handled in the request thread. It uses Celery queues:
- `ingest`
- `ocr`
- `extraction`
- `qdrant_check`
- `orchestration`
- `dlq`

The pipeline is:
1. `/contracts/upload` stores metadata and raw file, then enqueues `ingest_task`
2. `ingest_task` validates MIME/size and re-checks duplicates, then enqueues `ocr_task`
3. `ocr_task` runs OCR, caches OCR text in Redis, stores OCR text in Azure File Share, extracts parties, populates Neo4j parties/links, then enqueues `clause_extraction_task`
4. `clause_extraction_task` calls the LLM clause extractor, stores clauses in MongoDB, embeds clauses into tenant Qdrant collections, creates Neo4j clause nodes, then enqueues `qdrant_check_task`
5. `qdrant_check_task` compares each clause to playbook vectors in Qdrant
   - high-confidence matches are auto-approved and marked GREEN
   - lower-confidence matches are marked `ai_flagged` and sent to `review_orchestration_task`
6. `review_orchestration_task` runs a sequential LangGraph review pipeline per risky clause
7. Lawyers approve, reject, or request changes through `/clauses/*` endpoints
8. Completing a contract deletes raw file access artifacts while retaining review history

12. Key End-to-End Flows

12.1 Contract upload and processing
- user uploads file
- backend validates bytes, stores metadata, stores blob, and dispatches Celery
- OCR extracts text
- parties are heuristically extracted and linked in Neo4j
- clauses are extracted by LLM and stored
- clause vectors are embedded and indexed
- clauses are checked against playbook vectors
- risky clauses are sent to LangGraph review
- lawyer reviews results

12.2 Playbook ingestion
- admin uploads a PDF
- OCR returns markdown
- markdown is split by headings
- each chunk is sent to OpenAI for rule extraction
- valid rules are inserted into PostgreSQL
- each rule description is embedded into the playbook Qdrant collection

12.3 Clause decisioning
- lawyer approves, rejects, or requests changes
- Mongo clause state is updated
- Neo4j review relationship is written asynchronously in a background task

13. Important Implementation Risks and Drift

13.1 Test/code drift already present

`pytest` collection currently fails because:
- `tests/unit/test_neo4j_contract_graph.py` imports `graph_check_node`, which does not exist
- `tests/unit/test_risk_calculator.py` imports `contracts_platform.orchestration.scoring.risk_calculator`, which does not exist

This means the repository still contains tests for an older orchestration design that has been removed or renamed.

13.2 Empty modules and placeholders

Empty files were found in places that suggest unfinished or abandoned pieces:
- `contracts_platform/api/middleware/tracing_middleware.py`
- `contracts_platform/api/schemas/risk.py`
- several `__init__.py` files
- several integration test files are empty
- `scripts/generate_api_key.py` is empty
- `contracts_platform/db/neo4j/models.py` is empty

13.3 Status model is only partially consistent

Defined statuses include `OCR_COMPLETE` and `EXTRACTION_COMPLETE`, but worker tasks mostly set generic `PROCESSING` plus a `current_stage`. The finer-grained statuses exist in enums and frontend types but are not consistently emitted by workers.

13.4 Qdrant playbook semantics are inconsistent

Playbook rule vectors are inserted with payload fields like `rule_type`, `description`, and `source`, but `qdrant_check_task` searches the playbook store with `status="accepted"`, which playbook rules do not obviously carry as a semantic concept. Auto-approval logic may therefore depend on assumptions that are not clearly enforced by the playbook ingestion path.

13.5 Rule matching in PostgreSQL is simplistic

`playbook_score_node` checks:
- FORBIDDEN rules by substring inclusion
- REQUIRED rules by substring absence

This is easy to implement but weak for legal text. It can create false positives and false negatives because it does not reason semantically, only textually.

13.6 Frontend/backend user shape mismatch

The frontend `User` type expects `full_name`, and both `Sidebar` and `TopBar` render it, but backend user creation and auth responses only guarantee `email`, `role`, `tenant_id`, and IDs. Unless Mongo already stores `full_name`, the UI can render `undefined`.

13.7 Vite proxy and Axios base URL mismatch

The Vite config proxies `/api` and `/ws`, but the frontend Axios client uses `VITE_API_URL` directly and calls paths like `/auth/login` and `/contracts/*`. This works only if the env var is configured correctly; the proxy is not the main path used by the client.

13.8 Review completion logic is duplicated

Contract readiness is set in both:
- `run_review_graph`
- `review_orchestration_task`

The conditions are slightly different. That can cause status race conditions or confusing state transitions.

13.9 Redis temp collection key is read but not clearly written

`clause_extraction_task` reads `temp_collection:{contract_id}` from Redis and tries to clean it up, but the reviewed OCR path does not create or register that key. This suggests an unfinished RAG/temp-index pipeline.

13.10 Celery and event-loop design is pragmatic but brittle

The workers use sync Celery tasks with `asyncio.run()` wrappers. That works, but it increases complexity around DB clients and event-loop lifetimes, especially for SQLAlchemy and Motor.

15. Bottom Line

This project is a serious multi-service legal-tech platform, not a toy CRUD app. The strongest parts are:
- clear separation between API, workers, orchestration, and datastores
- thoughtful use of Mongo/PostgreSQL/Qdrant/Neo4j for distinct roles
- a usable lawyer-facing review UI
- strong administrative tooling for playbook management
